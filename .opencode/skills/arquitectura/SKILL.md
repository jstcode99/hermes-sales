# Architecture — Modular (Next.js App Router)

## Philosophy

**Domain-based modular** architecture: each module in `@modules/` is self-contained and groups
everything it needs (UI, logic, validation, data access). Shared code resides
in `@components/`, `@hooks/`, `@lib/`, and `@entities/`.

## Layers and dependency rule

```
app/              ← Routing only: pages, layouts, params, API routes
modules/<domain> ← Core: logic, data, UI, and validation per feature
components/       ← Reusable UI (base shadcn + shared components)
lib/              ← Global config (Supabase clients, utils)
hooks/            ← Global hooks
types/            ← Global types / generated by Supabase CLI
```

**Key rule:** `@app/` only imports from `@modules/app.modules.ts`. Modules can import from
`@components/`, `@lib/`, `@hooks/`, and `@types/`, but **never between different modules**

## Folders structure

```
/src
├── app/                                           # (Routing) Solo define rutas, layouts y recibe params
│   ├── (auth)/                                    # Grupos de rutas
│   ├── dashboard/
│   └── api/                                       # Webhooks o endpoints externos
├── components/                                    # UI Compartida
│   ├── ui/                                        # Componentes de shadcn (atómicos)
│   └── shared/                                    # Botones complejos, headers, etc.
├── modules/                                       # 🎯 EL NÚCLEO (Domain/Feature Layer)
│   ├── tickets/                                   # Ejemplo de un dominio
│   │   ├── components/components.tsx              # UI específica de tickets (DataTables, Forms)
│   │   ├── actions/ticket.actions.ts              # Server Actions (Equivalente al "Controller")
│   │   ├── services/ticket.services.ts            # Llamadas a Supabase (Equivalente al "Model/Repo")
│   │   ├── validations/tickets.validation.ts      # Validaciones Zod y tipos de TS
│   │   └── hooks.ts                               # Hooks específicos (si aplica)
│   └── users/                                     # Otro dominio...
├── lib/                                           # Configuraciones globales
│   ├── supabase/                                  # Clientes (client.ts, server.ts, admin.ts)
│   └── utils.ts                                   # Utils de tailwind/shadcn
├── hooks/                                         # Hooks globales (use-mobile, etc.)
└── types/                                         # Tipos globales o generados por Supabase CLI
```

## Anatomy of a Module

Each file within `@modules/tickets/` has a clear responsibility:

| File | Responsibility |
|---|---|
| `schema.ts` | Zod schemas for validation + inferred TypeScript types |
| `services.ts` | Read/query functions to Supabase (only `select`) |
| `actions.ts` | Server Actions for writes (`insert`, `update`, `delete`) |
| `hooks.ts` | Client hooks that consume actions/services (React Query, etc.) |
| `components/` | React components exclusive to this module |

## Where to Put New Items

| What you create | Where |
|---|---|
| New module/domain | `@modules/<domain>/` |
| Validation schema + types | `@modules/<domain>/validations/some.validation.ts` |
| Supabase queries (reads) | `@modules/<domain>/services/some.service.ts` |
| Mutations / Server Actions | `@modules/<domain>/actions/some.action.ts` |
| Module client hook | `@modules/<domain>/hooks/some.hook.ts` |
| Module UI component | `@modules/<domain>/components/<component>.tsx` |
| Shadcn component (atomic) | `@components/ui/<component>.tsx` |
| Shared component | `@components/shared/<component>.tsx` |
| Page / route | `@app/<route>/page.tsx` |
| Global hook | `@hooks/use-<name>.ts` |
| Global config / util | `@lib/utils.ts` or `@lib/<name>.ts` |
| Supabase client | `@lib/supabase/{client|server|admin}.ts` |
| Global types | `@types/index.ts` |
| Supabase generated types | `@types/database.types.ts` (do not edit manually) |
| SQL migration | `supabase/migrations/<timestamp>_<description>.sql` |

## When adding a new module, follow this order

1. Create the `@modules/<domain>/` folder
2. `schema.ts` → define Zod types + infer TypeScript types
3. `services.ts` → implement read queries using the Supabase client
4. `actions.ts` → implement write Server Actions, use schema for validation
5. `hooks.ts` → (if there is client-side logic) wrap with React Query or another library
6. `components/` → build the specific UI by consuming actions/services
7. `@app/<route>/page.tsx` → create the page and import components from the module

## Important Rules

- **`app/` contains no logic:** it only imports and renders components from `@modules/`.
- **Modules do not import from each other:** if two modules share logic, move it to `@lib/` or `@shared/`.
- **`services.ts` is read-only:** writes always go in `actions.ts` as Server Actions (`“use server”`).
- **`schema.ts` is the true source of types:** use `z.infer<typeof schema>` instead of defining individual types.
- **Supabase CLI:** When changing the DB schema, regenerate `@types/database.types.ts` with `supabase gen types typescript`.
